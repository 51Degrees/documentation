fn build_pipeline ( data_file:& Path) -> Arc< Pipeline >

Build the on-premise pipeline the benchmark detects through.

HighPerformance holds the data set in memory for the fastest detections, which is what a throughput measurement wants, and a single requested property (IsMobile) keeps detection resolving one component, the fastest realistic configuration. No ShareUsageElement is added, as usage sharing is forbidden for console/offline tools and would add network work that has nothing to do with the figure being measured. A production web deployment should enable it with .share_usage(true).

Run one detection per record, reading IsMobile so the optimizer cannot elide the work being measured, and return how many detections succeeded.

Per-detection errors are ignored: a single malformed record must not abort the benchmark, and only successful detections are counted.

◆ load_records()

fn load_records ( path:& Path,
max:usize  ) -> Vec< BTreeMap< String, String > >

Load up to max evidence records from the multi-document YAML evidence file.

The file is a ----separated stream, one document per record, each a map of evidence key (for example header.user-agent) to a string value. Each chunk is parsed on its own into a BTreeMap, mirroring the offline-processing example's reader. Records that fail to parse are skipped so one odd document cannot abort the load.

◆ RECORD_COUNT

const RECORD_COUNT

How many evidence records to load and replay per benchmark iteration.

The bundled file holds 20,000. A few hundred keeps each iteration short enough for Criterion to gather a stable sample while still exercising a realistic mix of User-Agents and User-Agent Client Hints.
